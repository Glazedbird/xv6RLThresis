\section{xv6 调度机制分析}

在第一章明确研究问题与总体目标之后，本章进一步聚焦本文的系统基础，即 xv6 操作系统中的进程管理与调度机制。首先介绍 xv6 的进程模型与进程状态转换机制；然后讨论其默认轮转调度思想及其局限性；最后从序列决策角度对调度问题进行形式化建模，为下一章的 Q-learning 算法设计提供系统背景与建模基础。

\subsection{xv6 进程管理机制}

\subsubsection{进程模型}

在 xv6 操作系统中，进程（process）是系统资源管理与任务调度的基本单位。xv6 采用较为简化的执行模型，不区分传统意义上的进程与线程，因此系统中的每个执行实体都以进程形式存在，并作为调度器进行 CPU 分配的直接对象。

xv6 通过一张全局进程表 \texttt{proc[]} 管理系统中的全部进程。该进程表本质上是一个固定大小的数组，每个数组元素对应一个进程控制块，用于维护进程在生命周期中的状态、上下文以及相关资源信息。系统通过对进程表项的分配、状态更新和回收，完成进程的创建、执行、阻塞、唤醒和退出等基本管理过程。

从资源组织的角度看，一个 xv6 进程通常包含以下几类核心资源：

\begin{itemize}
\item 用户地址空间与页表；
\item 内核栈；
\item 保存内核执行上下文的 \texttt{context} 结构；
\item 陷入内核时使用的 \texttt{trapframe}；
\item 打开文件表和当前工作目录等资源描述信息。
\end{itemize}

在实现层面，xv6 使用 \texttt{struct proc} 描述单个进程。该结构除了包含 \texttt{pid}、\texttt{state}、\texttt{parent}、\texttt{pagetable}、\texttt{context} 等基本字段外，在本文实现中还进一步扩展了创建时间、退出时间、运行时间、等待时间、首次运行时间和调度次数等统计字段，为后续调度建模和实验分析提供支持。

\subsubsection{进程状态与状态转换}

从运行状态角度看，xv6 中的进程可视为一个有限状态机。根据 \texttt{proc.h} 中的定义，进程主要包括以下几种状态：

\begin{itemize}
\item \texttt{UNUSED}：进程表项未被占用；
\item \texttt{USED}：进程控制块已分配，但尚未进入可运行状态；
\item \texttt{SLEEPING}：进程因等待某个事件而阻塞；
\item \texttt{RUNNABLE}：进程已具备运行条件，等待被调度；
\item \texttt{RUNNING}：进程当前正在 CPU 上执行；
\item \texttt{ZOMBIE}：进程已经退出，但尚未被父进程回收。
\end{itemize}

进程在其生命周期中会在上述状态之间不断转换。一般而言，进程表项从 \texttt{UNUSED} 被分配后进入 \texttt{USED}，随后在初始化完成后转入 \texttt{RUNNABLE}；调度器从 \texttt{RUNNABLE} 集合中选择进程后，将其切换为 \texttt{RUNNING}；进程在执行过程中可能由于主动让出处理器而回到 \texttt{RUNNABLE}，也可能因等待事件而进入 \texttt{SLEEPING}；等待条件满足后，系统通过唤醒机制使其重新回到 \texttt{RUNNABLE}；进程执行结束后则进入 \texttt{ZOMBIE}，最终在被父进程回收后重新变为 \texttt{UNUSED}。

这种状态转换机制构成了 xv6 调度器运行的基础。无论是默认的轮转调度，还是本文后续实现的 Q-learning 调度器，本质上都建立在对 \texttt{RUNNABLE} 集合的选择与对进程状态的动态更新之上。

\subsubsection{父子进程关系与资源回收}

xv6 使用父子进程关系管理进程之间的派生结构。当进程调用 \texttt{fork()} 时，系统为其创建子进程，并复制父进程的地址空间与部分执行环境。父进程可以通过 \texttt{wait()} 等待子进程结束，并在其退出后回收资源。

在本文实现中，除原有 \texttt{wait()} 外，还扩展了 \texttt{waitstat()} 接口，使父进程在等待子进程退出时，不仅能够获得退出状态，还可以获取调度统计信息。这一机制既保留了 xv6 原有的进程回收模式，也为实验章节的数据采集提供了基础支撑。

\subsubsection{上下文切换机制}

xv6 的上下文切换主要发生在内核态，通过保存和恢复不同进程的内核上下文来实现。每个进程的 \texttt{context} 保存其在内核中的寄存器状态，而 CPU 也维护一份调度器上下文。当当前进程因时间片结束、主动让出处理器、阻塞或退出而离开 CPU 时，系统会调用 \texttt{swtch()} 保存当前上下文，并切换到调度器上下文；调度器选定下一个进程后，再次调用 \texttt{swtch()} 恢复该进程的执行上下文。

这种切换机制的本质是执行流在“进程内核上下文”和“调度器上下文”之间来回转移。对于本文后续引入强化学习调度策略而言，调度器正是在这一切换点上获得决策机会。

\subsection{xv6 默认调度思想分析}

\subsubsection{轮转调度的基本思想}

xv6 原始系统采用轮转调度（Round-Robin）思想。在这一机制下，调度器不断扫描进程表，按固定顺序寻找处于 \texttt{RUNNABLE} 状态的进程，并使其获得一次 CPU 执行机会。定时器中断会周期性触发抢占，使正在运行的进程在适当时机让出处理器，从而保证不同任务都能在有限时间内获得执行机会。

轮转调度的核心特点在于其公平性与实现简洁性。由于所有可运行进程都按照近似相同的方式轮流获得 CPU，系统不需要维护复杂优先级或任务估计信息，因而十分适合作为教学操作系统中的基础调度方案。

\subsubsection{默认调度器的优点与局限}

Round-Robin 调度具有以下优点：

\begin{itemize}
\item 实现简单，便于在内核中维护；
\item 能够保证各可运行进程都获得执行机会，具有较好的基本公平性；
\item 不依赖任务运行时间预测或复杂历史信息。
\end{itemize}

但与此同时，其局限性也较为明显：

\begin{itemize}
\item 调度规则固定，缺乏对系统负载变化的适应能力；
\item 不区分长作业、短作业和阻塞型作业的行为差异；
\item 难以针对响应时间、等待时间或吞吐效率等目标进行有针对性的优化；
\item 不利用历史调度结果对未来决策进行修正。
\end{itemize}

从决策机制上看，默认轮转调度更接近一种基于固定规则的静态策略。它并不会依据系统当前状态学习或调整行为，因此在复杂、动态变化的工作负载下，其调度效果往往难以达到最优。

\subsection{调度问题的结构化建模}

随着强化学习技术的发展，越来越多的研究尝试将操作系统调度问题建模为序列决策问题，通过系统与环境之间的持续交互不断优化调度策略。对于调度器而言，每一次调度触发都可以看作一次决策时刻：系统观察当前状态，选择一种调度行为，并根据执行结果获得反馈。

形式化地，调度问题可以抽象为马尔可夫决策过程（Markov Decision Process, MDP）。在每个决策时刻，系统处于状态 $s$，调度器执行动作 $a$，系统转移到新状态 $s'$，并获得即时奖励 $r$。重复这一过程，即可将调度优化问题转化为策略学习问题。

标准 MDP 一般由四元组表示：

\begin{equation}
(S, A, P, R)
\end{equation}

其中：

\begin{itemize}
\item $S$ 表示状态空间；
\item $A$ 表示动作空间；
\item $P$ 表示状态转移关系；
\item $R$ 表示奖励函数。
\end{itemize}

需要指出的是，操作系统调度环境严格来说具有一定的不完全可观测性。例如，进程未来还需运行多长时间、后续是否会阻塞以及外部负载如何变化，通常都无法被调度器完全提前获知。因此，从严格意义上说，调度问题更接近部分可观测决策过程。然而，为了控制建模复杂度并便于在轻量级内核中实现，本文采用对系统状态进行近似编码的方式，将其简化为有限状态的 MDP，并在此基础上应用 Q-learning 进行求解。

\subsubsection{状态空间建模}

为了兼顾实现复杂度与状态表达能力，本文并未直接使用高维、连续的系统特征，而是从 xv6 调度器能够直接观测和维护的统计信息中提取若干关键量作为状态构成要素，例如可运行任务规模、等待压力以及反映调度历史的运行统计信息等。

在具体实现中，这些变量会被进一步离散化为有限数量的状态类别，并组合编码为整数状态编号。这样的处理方式一方面保留了系统负载与等待压力等关键调度信息，另一方面避免状态空间过大，从而便于后续采用表格型 Q-learning 进行在线学习。具体的状态定义与编码方式将在下一章结合算法设计统一给出。

\subsubsection{动作空间建模}

在传统调度问题中，动作通常可定义为“直接选择某个具体进程执行”。但在内核环境中，这种定义会导致动作空间规模随可运行进程数变化而变化，不利于使用表格型强化学习算法。

因此，本文采用策略级动作建模方式，将动作定义为“选择一种调度规则”。具体而言，调度器并不直接输出某个进程编号，而是先在若干候选规则之间进行选择，再由该规则在当前 \texttt{RUNNABLE} 集合中确定实际执行的进程。该处理方式能够显著降低动作空间维度，并提高学习过程的稳定性。至于候选规则的具体组成及其选择逻辑，将在下一章中结合 Q-learning 算法统一说明。

\subsubsection{奖励建模}

奖励函数用于衡量一次调度决策对系统运行效果的影响。在本文建模中，奖励设计主要围绕系统等待压力的变化展开，即通过比较调度前后系统状态的变化，对调度行为进行奖惩；同时，当一次调度推动进程完成时，也可给予额外正反馈。

这种奖励设计虽然较为简化，但能够在不引入复杂开销的前提下，为后续强化学习算法提供清晰的优化方向。具体的奖励函数形式及其参数设置将在下一章中展开。

\subsubsection{状态转移与无模型学习}

在标准 MDP 中，状态转移通常由 $P(s'|s,a)$ 描述。但在实际操作系统调度环境中，状态转移由时间片中断、进程执行行为、阻塞与唤醒等多种机制共同决定，难以显式建模。因此，本文并不试图解析给出状态转移概率，而是采用无模型（model-free）强化学习方式，通过不断采样交互序列 $(s,a,r,s')$ 直接学习调度策略。

这一建模方式与 xv6 当前实现高度契合。调度器在每次调度决策后都可以自然获得“调度前状态、所选动作、执行结果、调度后状态”这一完整样本，从而适合采用 Q-learning 进行在线更新。

\subsection{本章小结}

本章对 xv6 的进程管理与调度机制进行了分析，并从序列决策角度对调度问题进行了结构化建模。首先介绍了 xv6 的进程模型、状态转换机制和上下文切换过程；随后分析了默认轮转调度思想的优点与局限；最后给出了将调度问题抽象为有限状态 MDP 的思路，并说明了状态空间、动作空间、奖励机制以及无模型学习的基本建模方式。这些内容为下一章给出具体的 Q-learning 调度算法设计奠定了理论基础。
