\section{xv6调度机制分析}
本章对xv6操作系统中的进程调度机制进行分析。首先介绍xv6进程管理结构及管理结构，然后分析其默认的时间片轮转调度算法及其特点，最后从决策的角度对人物调度进行形式化建模，为后续基于强化学习的调度策略设计提供理论基础。
\subsection{xv6进程管理机制}
\subsubsection{进程模型}
在 xv6 操作系统中，进程（process）是系统资源管理与任务调度的基本单位。xv6 采用简化的进程模型，不区分进程与线程，因此系统中的每个执行实体都以进程的形式存在，并作为调度器进行 CPU 分配的基本对象。

xv6 通过一张全局进程表\texttt{proc[]} 来管理系统中的所有进程。该表本质上是一个固定大小的数组，每个元素对应一个进程控制块（Process Control Block, PCB）。操作系统通过维护该表中的状态信息，实现进程的创建、调度、阻塞与回收等生命周期管理。

在资源管理层面，进程是系统资源的主要持有者。一个进程通常包含以下几类资源：
\begin{itemize}
    \item 地址空间（page table）：描述进程的虚拟内存映射
    \item 寄存器上下文（register context）：保存 CPU 执行状态
    \item 内核栈（kernel stack）：用于系统调用与内核执行
    \item 打开文件表（file descriptors）：管理进程持有的文件资源
\end{itemize}

从操作系统运行状态的角度来看，进程可以被视为一个有限状态机。在其生命周期中，进程会在不同状态之间转换，以配合系统调度与资源管理。xv6 定义了以下几种进程状态：
\begin{itemize}
    \item UNUSED：进程表项未被使用
    \item USED：进程结构已分配但尚未运行
    \item RUNNABLE：进程可运行，等待 CPU 调度
    \item RUNNING：进程正在 CPU 上执行
    \item SLEEPING：进程因等待某事件而阻塞
    \item ZOMBIE：进程已退出但尚未被父进程回收
\end{itemize}
\begin{figure}[htbp]
    \centering
    \includegraphics[width=0.7\textwidth]{figures/xv6系统状态图.pdf}
    \caption{xv6系统状态图}
    \label{fig:proc_state}
\end{figure}

如图~\ref{fig:proc_state}所示，xv6操作系统中的进程在不同状态之间进行转换，
主要包括 UNUSED、USED、RUNNABLE、RUNNING、SLEEPING 和 ZOMBIE 等状态。
进程在创建后由 UNUSED 状态转为 USED，随后进入 RUNNABLE 状态等待调度。

调度器从 RUNNABLE 状态的进程中选择一个进程执行，使其进入 RUNNING 状态。
在运行过程中，进程可能因主动让出 CPU（如调用 yield）重新进入 RUNNABLE 状态，
也可能因等待某些事件（如 I/O 操作）进入 SLEEPING 状态。
当等待的事件发生时，通过唤醒机制（wakeup）使进程重新回到 RUNNABLE 状态。

当进程执行结束后，将进入 ZOMBIE 状态等待父进程回收资源，
最终释放后重新回到 UNUSED 状态，从而完成一次完整的生命周期。

在进程组织方式上，xv6 采用 父子进程（parent-child）结构 来管理进程关系。当一个进程调用 \texttt{fork()} 创建新进程时，新进程成为其子进程，并继承部分资源信息。父进程可以通过 \texttt{wait()} 系统调用等待子进程结束并回收其资源，这一机制与 Linux 等现代操作系统的基本进程模型类似。

在实现层面，xv6 使用 \texttt{struct proc} 结构体来描述每个进程的运行状态及其相关信息。该结构体包含：
\begin{itemize}
    \item 进程标识符 pid
    \item 进程状态 state
    \item 内核栈指针
    \item 页表 pagetable
    \item 进程上下文 context
    \item 打开文件表
    \item 父进程指针等信息
\end{itemize}

其中，\texttt{context} 结构用于保存进程在 内核态执行时的寄存器上下文。当调度器发生进程切换时，系统通过 \texttt{swtch()} 函数保存当前进程的寄存器状态，并恢复目标进程的上下文，从而实现不同进程之间的执行切换。

通过上述机制，xv6 实现了一个简洁而完整的进程管理与调度框架，为后续调度算法的实现与扩展提供了基础。

\subsubsection{调度循环}
\begin{figure}[htbp]
    \centering
    \includegraphics[width=0.7\textwidth]{figures/scheduler调度流程图.pdf}
    \caption{调度循环流程图}
    \label{fig:scheduler_flow}
\end{figure}

如图~\ref{fig:scheduler_flow}所示，xv6操作系统中的调度器以循环方式运行，
在每个CPU上持续执行调度流程。调度器首先遍历进程表 proc[]，查找处于 RUNNABLE 状态的进程。

若在当前扫描过程中未找到可运行进程，则调度器会检查系统是否存在中断或
其他唤醒事件；若没有可处理事件，则进入等待状态，直到新的中断发生后再次
进入调度循环。

当找到 RUNNABLE 进程时，调度器选择其中一个进程 p，将其状态设置为
RUNNING，并通过上下文切换函数 swtch() 将 CPU 的执行权交给该进程。
进程获得 CPU 后开始执行，直至发生时间片耗尽、主动让出处理器（yield）、
进入休眠（sleep）或执行结束（exit）等情况。

当上述事件发生时，控制权重新返回调度器，调度器再次进入扫描与决策过程，
从而形成一个不断重复的调度循环。该循环体现了操作系统中基于状态检测与
事件驱动的调度机制。

\subsubsection{上下文切换机制}
如图~\ref{fig:swtch}所示，xv6操作系统中的上下文切换主要发生在内核空间中，
通过内核栈（kstack）保存与恢复进程的执行上下文。

当当前进程（如 shell）主动让出 CPU 或被调度器切换时，系统首先将该进程的寄存器
状态保存到其对应的内核栈中（save）。随后通过 swtch() 函数，将执行流切换到
调度器的内核栈（kstack scheduler），此时 CPU 开始在调度器上下文中运行。

调度器根据调度策略选择下一个要执行的进程（如 cat），并再次调用 swtch()，
将执行流从调度器切换到目标进程的内核栈（kstack cat）。在此过程中，系统从该
进程的内核栈中恢复其之前保存的寄存器状态（restore），从而使该进程从之前中断
的位置继续执行。

整个过程体现了上下文切换的本质：通过保存当前执行上下文并恢复目标上下文，
在不同进程之间切换CPU的执行权，而无需改变CPU本身的运行机制。

\subsection{默认算法分析}
\subsubsection{Round Robin原理}
\begin{figure}[htbp]
    \centering
    \includegraphics[width=0.7\textwidth]{figures/RR调度流程图.pdf}
    \caption{Round Robin算法流程图}
    \label{fig:RR_flow}
\end{figure}
如图~\ref{fig:RR_flow}所示，Round Robin（时间片轮转）调度算法通过维护
一个循环就绪队列，实现对进程的公平调度。系统按照固定顺序遍历队列，
每次选择队首进程并分配一个固定长度的时间片进行执行。

当进程获得CPU后，其执行过程存在两种可能情况：一是进程在时间片内完成
执行或进入阻塞状态（如调用sleep()或发生I/O等待），此时该进程将从就绪
队列中移除，不再参与后续调度；二是时间片耗尽但进程仍未完成执行，此时
在定时器中断触发下，系统会强制剥夺CPU，并将该进程重新加入队列尾部，
等待下一轮调度。

\subsubsection{算法优缺点}
Round Robin 调度算法具有实现简单、公平性较好的特点，能够保证每个进程在有限时间内获得 CPU 资源，因此被广泛应用于教学操作系统和早期多任务系统中。
尽管时间片轮转调度具有较好的公平性，但该算法采用固定策略进行任务调度，不考虑任务执行特征、系统负载变化以及历史调度信息。因此在复杂计算环境下，其调度效率可能难以达到最优。
\begin{table}[htbp]
\centering
\caption{Round Robin调度算法特性分析}
\begin{tabular}{ccc}
\hline
方面 & 优点 & 局限性 \\
\hline
公平性 & 所有进程均可获得CPU时间 & 无法区分任务重要性 \\
实现复杂度 & 实现简单，开销较低 & 缺乏优化能力 \\
适应性 & 对简单负载表现稳定 & 无法适应动态负载变化 \\
决策依据 & 固定轮转顺序 & 不利用历史或状态信息 \\
\hline
\end{tabular}
\end{table}

从决策角度来看，Round Robin算法本质上是一种基于固定规则的静态调度策略，其决策过程不依赖于系统当前状态，也不利用历史信息进行优化。因此，可以将其视为一种无状态（stateless）的调度策略。
相比之下，在动态系统环境中，调度问题更适合建模为一个基于状态的序列决策过程，从而引出基于强化学习的调度方法。

\subsection{调度问题的结构建模}
随着强化学习技术的发展，越来越多研究尝试将操作系统调度问题建模为序列决策问题，通过与环境的交互不断优化调度策略。在该框架下，可以将每一次调度触发视为一个决策时刻：系统根据当前状态选择一个待执行进程，并依据调度结果获得反馈，从而逐步学习最优调度策略。

形式化地，调度问题可以抽象为一个\textbf{马尔可夫决策过程（Markov Decision Process, MDP）}。在每个决策时刻，系统处于状态 $s$，调度器选择动作 $a$（即选择一个进程执行），系统转移到新状态 $s'$ 并获得奖励 $r$，通过不断重复该过程完成策略优化。

马尔可夫决策过程的核心假设是马尔可夫性质（Markov Property），即系统的未来状态仅依赖于当前状态和当前动作，而与历史状态无关。形式化地，对于任意时间步 $t$，有：
\begin{equation}
P(s_{t+1} \mid s_t, a_t, s_{t-1}, a_{t-1}, \dots)
= P(s_{t+1} \mid s_t, a_t)
\end{equation}
该性质意味着当前状态 $s_t$ 已经包含了对未来决策所需的全部信息，因此可以基于当前状态进行最优决策。

需要指出的是，操作系统调度环境通常具有不完全可观测性。例如，
任务的未来运行时间、I/O行为以及外部负载变化均难以准确预测，
因此严格意义上该问题更符合\textbf{部分可观测马尔可夫决策过程（POMDP）}。

为降低问题复杂度并便于算法实现，本文对系统状态进行近似建模。
具体而言，引入进程的运行时间、等待时间等统计特征，对历史信息进行编码，使得当前状态能够在一定程度上反映系统的演化趋势，从而在实践中近似满足马尔可夫性假设。在此基础上，将调度问题简化为MDP进行建模与求解。

MDP通常由四个元素组成：
$$(S, A, P, R)$$
其中：
\begin{itemize}
    \item S：状态空间（State）
    \item A：动作空间（Action）
    \item P：状态转移概率（Transition）
    \item R：奖励函数（Reward）
\end{itemize}
在操作系统调度问题中，可以对上述元素进行具体定义。
\subsubsection{状态空间}
状态空间用于描述当前系统的运行情况。在任务调度场景中，系统状态可以由多个指标共同构成，例如当前可运行进程数量、各进程的等待时间、运行时间以及历史调度信息等。

在本文的设计中，为了兼顾系统信息表达能力与实现复杂度，状态主要由以下指标构成：
\begin{itemize}
    \item $n$：当前 \textbf{RUNNABLE 进程数量}
    \item $w$： 进程的 \textbf{等待时间（wait time）}
    \item $r$：进程的 \textbf{累计运行时间（run time）}
    \item $c$：进程的 \textbf{调度次数（schedule count）}
\end{itemize}
这些信息可以从 xv6 的 struct proc 结构体中统计得到，并能够反映进程在系统中的运行状态。通过将这些指标组合，可以构建当前调度时刻的系统状态表示。
形式化表示为：
$$s=(n,w,r,c) \in S$$
其中$s$代表系统当前状态。
\subsubsection{动作空间}
在操作系统调度问题中，调度器在每一次调度时刻需要决定选择哪个进程运行。因此动作空间可以定义为：
$$A = 选择某个RUNNABLE进程$$
当系统存在$N$个可运行进程时，动作空间大小为：
$$|A| = N$$
调度器通过选择不同的动作，可以将 CPU 分配给不同进程，从而影响系统运行性能。
\subsubsection{奖励函数}
奖励函数用于评价调度行为对系统性能的影响，是强化学习模型设计中的关键部分。合理的奖励设计可以引导算法学习到更优的调度策略。

在操作系统调度问题中，通常关注以下性能指标：
\begin{itemize}
    \item \textbf{响应时间（Response Time）}
    \item \textbf{等待时间（Waiting Time）}
    \item \textbf{系统吞吐量（Throughput）}
\end{itemize}
为了鼓励调度策略减少进程等待时间并提高系统整体效率，可以设计奖励函数，使得等待时间较短或调度效率较高时获得更高奖励。
例如，可以定义奖励为：
$$r = - waiting\_time$$
或根据多个指标进行组合：
$$r = -\alpha \times waiting\_time + \beta \times throughput$$
其中$\alpha$和$\beta$为权重参数。
通过这种方式，调度器能够在不断交互中学习到能够减少等待时间并提高系统效率的策略。

\subsection{状态转移建模}

在标准马尔可夫决策过程（MDP）中，状态转移概率 $P(s'|s,a)$ 用于描述在状态 $s$ 下执行动作 $a$ 后转移至下一状态 $s'$ 的概率分布。

然而，在实际操作系统调度场景中，该转移过程由系统运行机制决定，涉及进程执行、时间片调度、中断处理等复杂因素，难以进行显式建模。因此，本文不对转移概率进行解析建模，而是采用基于交互采样的方式进行学习。

具体而言，在调度过程中，每执行一次调度决策即可获得一组状态转移样本 $(s,a,r,s')$，强化学习算法基于该样本对价值函数进行更新。该方法属于无模型强化学习（model-free reinforcement learning）范式，不依赖于转移概率的显式表达。

因此，本文所构建的 MDP 模型中，状态转移函数由操作系统运行环境隐式定义，而强化学习算法通过与系统交互逐步逼近最优策略。

\subsubsection{调度问题的MDP建模}

在上述定义下，操作系统调度过程可以形式化建模为一个马尔可夫决策过程（MDP）。调度决策以时间片为单位进行，在每一个调度时刻（decision epoch），系统按照如下流程运行：

\begin{enumerate}
    \item 系统处于当前状态 $s$，该状态由系统运行时的统计信息构成；
    \item 调度器根据当前策略 $\pi$ 选择动作 $a$；
    \item 动作 $a$ 对应一种调度规则，由该规则在 RUNNABLE 进程集合中选取具体进程执行；
    \item 系统运行所选进程一个时间片，并根据执行结果更新系统状态至 $s'$；
    \item 根据状态变化及执行结果计算即时奖励 $r$。
\end{enumerate}

通过不断重复上述过程，系统能够产生一系列状态转移样本 $(s,a,r,s')$。强化学习算法基于这些交互数据对策略进行更新，从而逐步逼近最优调度策略。

需要指出的是，本文未对状态转移概率 $P(s'|s,a)$ 进行显式建模，而是采用无模型强化学习（model-free reinforcement learning）方法，通过与操作系统环境的交互采样实现策略学习。

在具体实现中，本文采用 Q-learning 算法对动作价值函数 $Q(s,a)$ 进行估计，并将学习过程嵌入 xv6 操作系统调度器中，实现基于强化学习的自适应调度机制。
该建模方式将操作系统调度问题从传统启发式策略选择问题转化为序贯决策优化问题，从而为调度策略的自适应优化提供了统一框架。
