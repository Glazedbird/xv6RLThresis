\section{实验设计与性能评估}

为了验证基于 Q-learning 的调度算法在 xv6 任务调度场景中的可行性，本文在 xv6 操作系统上实现了 QL-Scheduler，并结合用户态工作负载程序对其运行行为进行观测与评估。本章首先介绍实验环境与测试方法，然后结合当前工程实现说明实验负载、观测指标与结果分析方法。

\subsection{实验环境}

实验平台基于 xv6 操作系统，并在 QEMU 虚拟机环境中运行。系统环境如下：

\begin{itemize}
\item 模拟器：QEMU；
\item 操作系统：xv6 (RISC-V)；
\item 编译工具链：\texttt{riscv64-unknown-elf-gcc}；
\item 运行方式：在 xv6 用户态执行测试程序，由内核 \texttt{waitstat()} 接口回收进程统计信息。
\end{itemize}

实验过程中，内核在进程控制块中维护创建时间、退出时间、累计运行时间、累计等待时间、首次运行时间与被调度次数等统计字段；用户态通过 \texttt{workload} 程序创建不同类型的任务组合，并在任务结束后汇总输出平均周转时间、平均响应时间、平均等待时间、平均运行时间以及平均调度次数等指标。

\subsection{实验负载设计}

本文的实验负载由用户程序 \texttt{workload.c} 统一生成。该程序通过 \texttt{fork()} 与 \texttt{exec()} 创建多组子进程，并利用 \texttt{waitstat()} 回收各子进程的运行统计信息。按照当前工程实现，实验共设计了 4 类工作负载模式。

\subsubsection{模式 0：长 CPU 密集型负载}

在模式 0 下，\texttt{workload} 会连续创建 4 个 \texttt{cpu\_long} 任务。该任务通过大规模整数累加循环持续占用处理器，几乎不主动休眠，可用于观察调度器在同构长作业竞争场景下的行为。

\subsubsection{模式 1：短 CPU 密集型负载}

在模式 1 下，\texttt{workload} 会创建 8 个 \texttt{cpu\_short} 任务。相较于 \texttt{cpu\_long}，该任务计算规模更小，执行时间更短，可用于观察调度器在大量短作业并发到达时对响应性和完成效率的影响。

\subsubsection{模式 2：长短混合 CPU 负载}

在模式 2 下，\texttt{workload} 会创建 2 个 \texttt{cpu\_long} 任务和 6 个 \texttt{cpu\_short} 任务。该模式对应典型的长短作业混合场景，可用于分析调度器在公平性与短作业优先倾向之间的平衡能力。

\subsubsection{模式 3：计算与阻塞混合负载}

在模式 3 下，\texttt{workload} 会创建 1 个 \texttt{cpu\_long} 任务以及 4 个 \texttt{sleepy\_job} 任务。\texttt{sleepy\_job} 在每轮短时间计算后调用 \texttt{pause(10)} 主动休眠，因此该模式可以近似反映“计算型任务与阻塞型任务并存”的调度场景，更有利于观察调度器对交互型任务响应性的影响。

\subsection{对比对象}

结合当前工程实现，本文实验中的主要对比对象为以下两类调度策略：

\begin{itemize}
\item \textbf{Round Robin (RR)}：xv6 默认的轮转调度策略，也是本文最直接的基线；
\item \textbf{QL-Scheduler}：本文实现的基于 Q-learning 的调度器。
\end{itemize}

需要说明的是，当前工程实现中并未同时实现可直接切换的 SJF 或静态优先级调度器，因此本章实验分析主要围绕原始 xv6 轮转调度与 QL-Scheduler 两者展开。这样既与当前实现保持一致，也有利于将强化学习调度器的效果直接归因于调度决策机制的变化。

\subsection{实验指标}

为了评估调度器在不同工作负载下的表现，本文选取当前工程中能够直接观测并稳定采集的指标进行分析，主要包括以下几项。

\begin{itemize}
\item \textbf{平均等待时间（Average Waiting Time）}

表示进程在 \texttt{RUNNABLE} 状态下累计等待 CPU 的平均时长。在本文实现中，该指标对应内核统计字段 \texttt{m\_wait\_ticks}。

\item \textbf{平均周转时间（Average Turnaround Time）}

表示进程从创建到退出的总经历时间，可由退出时刻与创建时刻之差计算得到，即
\begin{equation}
T_{\text{turnaround}} = e\_time - c\_time
\end{equation}

\item \textbf{平均响应时间（Average Response Time）}

表示进程从创建到首次获得 CPU 执行之间的时间，可由首次运行时刻与创建时刻之差计算得到，即
\begin{equation}
T_{\text{response}} = first\_run\_time - c\_time
\end{equation}

\item \textbf{平均运行时间（Average Running Time）}

表示进程实际处于 \texttt{RUNNING} 状态的累计时钟滴答数，在本文实现中对应统计字段 \texttt{m\_run\_ticks}。该指标主要用于辅助分析不同负载模式下任务本身的计算强度。

\item \textbf{平均调度次数（Average Schedule Count）}

表示进程在生命周期内被调度器选中的次数，在本文实现中对应字段 \texttt{m\_sched\_count}。该指标有助于观察调度器在不同负载下对时间片分配与进程切换频率的影响。
\end{itemize}

需要指出的是，虽然工程中保留了 \texttt{m\_sleep\_ticks} 字段，但当前版本尚未在时钟中断统计逻辑中对 \texttt{SLEEPING} 状态进行累计，因此该字段暂不作为本章的核心评价指标。同时，CPU 利用率也未在当前实验框架下形成统一、稳定的采集方式，因此本章不将其作为主要比较对象。

\subsection{实验数据获取方式}

实验数据由用户态程序 \texttt{workload} 自动汇总输出。对于每一种负载模式，\texttt{workload} 在所有子进程退出后，会基于 \texttt{waitstat()} 返回的统计结果计算如下平均值：

\begin{itemize}
\item 平均周转时间；
\item 平均响应时间；
\item 平均等待时间；
\item 平均运行时间；
\item 平均调度次数。
\end{itemize}

程序最终以一行 \texttt{SUMMARY} 结果输出所有汇总数据，因此可以通过重复运行不同模式，分别记录 RR 和 QL-Scheduler 下的实验结果，并据此开展横向对比分析。

\subsection{实验结果分析}

结合当前调度器实现与工作负载设计，本文的实验分析主要关注以下几个方面。

\subsubsection{长 CPU 密集型负载分析}

在模式 0 中，所有任务均为运行时间较长的 CPU 密集型作业，任务之间的差异较小。在这种场景下，QL-Scheduler 的优势通常不会表现为极大幅度缩短某一单项指标，而更多体现在对等待时间和调度次数分布的调节能力上。由于动作空间中包含“等待时间最长优先”“调度次数最少优先”和“轮转调度”等多种候选规则，调度器能够在保持基本公平性的同时，对长时间未被执行的任务给予一定补偿。

\subsubsection{短 CPU 密集型负载分析}

在模式 1 中，系统中同时存在较多短作业。此时，若调度策略能够更快地让短任务获得首次执行机会并尽早完成，则通常会带来更低的平均响应时间和平均周转时间。QL-Scheduler 的动作空间中包含“累计运行时间最短优先”规则，因此在部分状态下更可能偏向尽快处理短作业，这也是其相较纯轮转调度可能取得优势的重要原因。

\subsubsection{长短混合负载分析}

模式 2 同时包含长作业与短作业，是本文较为关键的评测场景。在该场景下，单纯轮转调度虽然具有较好公平性，但容易使短作业与长作业经历相近的轮转等待过程，从而削弱短作业应有的快速完成优势。QL-Scheduler 则可以根据系统中可运行任务数量与最大等待时间的变化，在不同启发式规则之间切换，因此有可能在不显著破坏公平性的前提下改善短作业的响应时间与平均完成效率。

\subsubsection{阻塞混合负载分析}

模式 3 同时包含持续计算任务和周期性休眠任务。对该类场景而言，调度器除了关注总体完成效率外，还需要兼顾被唤醒任务的及时响应。当前 QL-Scheduler 的奖励函数会根据全局最大等待时间桶的变化调整 Q 值，因此在阻塞型任务被唤醒后，若系统等待压力升高，调度器更可能倾向于采取有助于降低等待时间的动作。这使得其在混合负载下更有机会改善交互型任务的响应表现。

\subsection{结果解释与局限性}

从当前工程实现出发，QL-Scheduler 的优势主要来源于两点：其一，调度器不再固定使用单一规则，而是能够在四种候选动作之间进行自适应选择；其二，调度器在每次调度后根据状态转移结果更新 Q 表，从而逐步形成与系统负载相适应的策略偏好。

但与此同时，当前实现仍然存在一定局限性。首先，状态空间仅由“可运行进程数”和“最大等待时间”两个离散特征构成，表达能力仍较为有限；其次，奖励函数主要围绕等待压力变化进行设计，尚未显式纳入周转时间、响应时间等更丰富的优化目标；再次，\texttt{m\_sleep\_ticks} 尚未被完整统计，因此对阻塞行为的量化分析仍然较弱。上述因素决定了当前系统更适合作为强化学习调度器的原型验证，而非完整的生产级调度方案。

\subsection{本章小结}

本章结合当前工程实现，给出了实验环境、负载构造方式、对比对象、评价指标以及结果分析思路。与实现保持一致的实验框架表明，本文主要通过 \texttt{workload} 程序构造 4 类典型负载，并利用 \texttt{waitstat()} 收集等待时间、响应时间、周转时间、运行时间和调度次数等统计数据，以比较 QL-Scheduler 与原始 xv6 轮转调度在不同场景下的行为差异。这些实验设计为后续进一步补充定量结果和图表分析提供了基础。
