\section{基于 xv6 的 QL-Scheduler 实现}

第三章已经给出了基于 Q-learning 的调度器设计思想，本章进一步说明该方法如何在 xv6 内核中落地实现。与前文偏重建模和算法分析不同，本章关注工程实现层面的问题，即如何在尽量保持 xv6 原有结构清晰、改动范围可控的前提下，将状态构造、动作选择、奖励计算、Q 表更新以及实验观测接口嵌入到调度路径中，最终形成一个能够在线学习的内核级调度原型。

\subsection{总体实现思路}

xv6 原始调度器的核心位于 \texttt{scheduler()} 中，其基本行为是不断扫描进程表，从所有处于 \texttt{RUNNABLE} 状态的进程中选择一个投入运行。该实现结构简单，但策略固定，无法根据系统负载变化和历史调度结果动态调整。

本文的实现思路并不是直接让强化学习模块“在所有进程中挑一个进程”，而是将调度决策拆分为两个层次：

\begin{enumerate}
\item 先由强化学习模块根据当前全局状态选择一个动作；
\item 再由该动作对应的一条启发式规则，从 \texttt{RUNNABLE} 进程集合中选出具体的目标进程。
\end{enumerate}

这种设计有两个直接好处。其一，动作空间规模固定且较小，便于在内核中用有限 Q 表实现在线学习；其二，具体进程选择仍由内核中的确定性规则完成，不需要为每个进程单独建立高维状态表示，从而降低了实现复杂度。

结合代码实现，调度主循环可概括为以下过程：

\begin{enumerate}
\item 调用 \texttt{build\_global\_state()} 统计当前系统状态并完成离散编码；
\item 调用 \texttt{choose\_action(state)} 按照 $\varepsilon$-greedy 策略选择调度动作；
\item 调用 \texttt{pick\_proc\_by\_action(action)} 按照所选规则挑选目标进程；
\item 执行上下文切换，运行该进程一个时间片，或直到其主动阻塞、让出处理器或退出；
\item 进程返回调度器后，再次构造新状态；
\item 调用 \texttt{compute\_reward()} 计算奖励，并通过 \texttt{rl\_update()} 更新 Q 表。
\end{enumerate}

因此，一次完整的调度循环恰好对应一次强化学习中的交互步骤 $(s,a,r,s')$。这也是 Q-learning 模型与 xv6 调度路径结合的核心接口。

\subsection{相关数据结构设计}

\subsubsection{进程统计字段扩展}

调度器要完成状态建模和效果评估，首先需要从进程层面持续收集运行统计信息。为此，本文在 \texttt{struct proc} 中新增了若干字段，主要包括：

\begin{itemize}
\item \texttt{c\_time}：进程创建时刻；
\item \texttt{e\_time}：进程退出时刻；
\item \texttt{m\_run\_ticks}：进程处于 \texttt{RUNNING} 状态时累计获得的时钟滴答数；
\item \texttt{m\_wait\_ticks}：进程处于 \texttt{RUNNABLE} 状态时累计等待的时钟滴答数；
\item \texttt{m\_sleep\_ticks}：为睡眠时间统计预留的字段；
\item \texttt{first\_run\_time}：进程首次真正获得 CPU 的时刻；
\item \texttt{m\_sched\_count}：进程被调度器选中的次数；
\item \texttt{m\_last\_scheduled\_tick}：最近一次被调度的时刻；
\item \texttt{rl\_state}：为强化学习相关观测预留的状态字段。
\end{itemize}

这些字段分别在不同位置维护。进程创建时，\texttt{allocproc()} 记录 \texttt{c\_time}；进程退出时，\texttt{kexit()} 记录 \texttt{e\_time}；当某进程第一次被调度运行时，\texttt{scheduler()} 记录 \texttt{first\_run\_time}。与此同时，定时器中断处理函数 \texttt{clockintr()} 会周期性调用 \texttt{update\_sched\_stats()}，将当前正在运行的进程的 \texttt{m\_run\_ticks} 加一，并将所有处于 \texttt{RUNNABLE} 状态进程的 \texttt{m\_wait\_ticks} 加一。

需要说明的是，虽然结构体中保留了 \texttt{m\_sleep\_ticks} 字段，但当前版本的 \texttt{update\_sched\_stats()} 并没有对 \texttt{SLEEPING} 状态进程进行累计，因此该字段主要用于后续扩展，而不是本章实验中的核心观测量。

\subsubsection{Q 表与定点数实现}

Q-learning 需要维护状态动作价值函数 $Q(s,a)$。考虑到 xv6 内核环境较为简化，不适合引入复杂容器，本文采用静态二维数组保存 Q 表：

\begin{verbatim}
int qtable[NSTATE][NACTION];
struct spinlock qtable_lock;
\end{verbatim}

其中，\texttt{NSTATE} 表示离散状态总数，\texttt{NACTION} 表示动作总数。由于调度器和 Q 表可能在多核环境下被并发访问，因此代码中通过 \texttt{qtable\_lock} 保护 Q 表读取和更新，避免竞争条件。

此外，xv6 内核中通常不直接使用浮点运算，而标准 Q-learning 更新公式中的学习率 $\alpha$ 与折扣因子 $\gamma$ 一般为小数。因此，本文采用定点数方法实现：

\begin{itemize}
\item \texttt{SCALE = 1000}；
\item \texttt{ALPHA = 100}，对应 $\alpha = 0.1$；
\item \texttt{GAMMA = 900}，对应 $\gamma = 0.9$。
\end{itemize}

于是，Q-learning 更新过程中的实数运算被缩放为整数运算。若将真实价值记为 $\hat{Q}$，代码中存储的整数值记为 $Q = \hat{Q} \times \texttt{SCALE}$，则更新公式可写为：

\begin{equation}
Q(s,a) \leftarrow Q(s,a) + \frac{\texttt{ALPHA}\left(Target - Q(s,a)\right)}{\texttt{SCALE}}
\end{equation}

其中：

\begin{equation}
Target=
\begin{cases}
r \times \texttt{SCALE}, & \text{若为终止状态} \\
r \times \texttt{SCALE} + \dfrac{\texttt{GAMMA}\max_{a'}Q(s',a')}{\texttt{SCALE}}, & \text{否则}
\end{cases}
\end{equation}

这种做法避免了浮点支持问题，也使得整个强化学习模块能够以纯整数逻辑运行在 xv6 内核中。

\subsection{Q表持久化机制设计与实现}

在强化学习调度过程中，Q表用于存储状态—动作对的价值估计，是调度策略学习的核心数据结构。
若Q表仅存储于内存中，则在系统重启后学习结果将完全丢失，需要重新训练，
这将显著降低系统的实用性与稳定性。因此，本文设计并实现了Q表的持久化存储机制，
以支持学习结果的跨运行周期保留。

\subsubsection{设计思路}

Q表持久化的核心目标是：在保证系统稳定性的前提下，实现Q表在内存与磁盘之间的可靠同步。
结合xv6文件系统的结构特点，本文采用如下设计：

\begin{itemize}
    \item 将Q表作为一个普通文件存储于文件系统中；
    \item 使用inode层接口（如create、readi、writei）进行读写操作；
    \item 通过日志机制（log）保证写入的原子性与一致性；
    \item 引入版本号与校验机制确保数据合法性。
\end{itemize}

xv6文件系统采用分层设计，包括buffer cache、inode层以及日志层等结构为数据持久化提供了基础支持。

\subsubsection{存储数据结构设计}

为便于在磁盘中存储与恢复Q表，本文定义了专用的磁盘结构：

\begin{itemize}
    \item magic字段：用于标识数据合法性；
    \item version字段：支持未来扩展与兼容；
    \item nstate与naction：记录Q表维度；
    \item q数组：存储具体Q值。
\end{itemize}

内存中二维Q表通过“序列化”映射到该数据结构中，以连续内存的方式存储在磁盘中，便于一次性读写，提高I/O效率。

\subsubsection{持久化写入流程}

Q表写入过程如算法~\ref{alg:qsave} 所示，主要包括以下步骤：

\begin{algorithm}[H]
\caption{Q表持久化写入流程}
\label{alg:qsave}
\begin{algorithmic}[1]
\Require 内存中的 Q 表 \texttt{qtable}，磁盘结构体 \texttt{qtable\_disk}
\Ensure 将当前 Q 表内容写入磁盘文件

\State 获取 \texttt{qtable\_lock} 并将内存中的 Q 表拷贝至 \texttt{qtable\_disk}
\State 调用 \texttt{begin\_op()} 开启日志事务
\State 通过 \texttt{namei()} 或 \texttt{create()} 获取目标 inode，记为 $ip$
\If{$ip = 0$}
    \State 返回错误
\EndIf
\State 调用 \texttt{ilock()} 加锁目标 inode
\State 调用 \texttt{writei()} 将 \texttt{qtable\_disk} 写入磁盘
\State 调用 \texttt{iunlockput()} 释放 inode
\State 调用 \texttt{end\_op()} 提交日志事务
\State 返回写入结果
\end{algorithmic}
\end{algorithm}

具体而言，首先对内存中的 Q 表加锁，确保在复制过程中不会受到调度更新的并发干扰，然后将其内容拷贝至预定义的磁盘结构中。随后调用 \texttt{begin\_op()} 开启一次日志事务，以保证后续写操作的原子性。

在文件定位阶段，通过 \texttt{namei()} 查找既有文件，若文件不存在则通过 \texttt{create()} 创建对应 inode，从而建立 Q 表在文件系统中的存储载体。获得 inode 后，使用 \texttt{ilock()} 对其加锁，确保写入期间文件状态的一致性，并通过 \texttt{writei()} 将 Q 表数据写入磁盘。

写入完成后，通过 \texttt{iunlockput()} 释放 inode 并更新引用计数，最后调用 \texttt{end\_op()} 提交日志事务，使数据正式落盘。

该流程充分利用 xv6 的日志文件系统机制，在保证并发安全的同时，实现了 Q 表状态的可靠持久化，为强化学习调度策略的长期收敛提供了基础支持。

\subsubsection{持久化加载流程}

系统启动阶段，通过 Q 表加载函数从磁盘恢复历史学习结果，其整体流程如下：

\begin{enumerate}
\item 查找 Q 表文件对应的 inode；
\item 读取磁盘数据至内存结构；
\item 校验 magic、version 及维度信息；
\item 若校验通过，则更新内存 Q 表；
\item 若校验失败，则初始化默认 Q 表并重新写入磁盘。
\end{enumerate}

在具体实现中，加载过程发生于文件系统初始化完成之后，以确保 inode、buffer cache 及日志模块均已处于可用状态。通过 \texttt{namei()} 获取目标文件对应的 inode 后，利用 \texttt{readi()} 将磁盘中的 Q 表数据读入内存缓冲区。

为避免由于系统异常（如崩溃、中断写入）导致的数据损坏，设计了基于 \texttt{magic} 与版本号的完整性校验机制。其中，\texttt{magic} 用于识别数据结构是否合法，\texttt{version} 用于支持后续结构扩展，维度信息则用于保证 Q 表与当前内核配置一致。只有在三者均匹配的情况下，才将数据加载至内存中的 Q 表结构。

若校验失败，则认为磁盘数据不可用，系统将回退至默认初始化策略，并触发一次写回操作，以重建合法的 Q 表文件。该机制保证了系统在异常情况下仍可恢复到可运行的初始状态，从而提高整体鲁棒性。

\subsubsection{并发与一致性控制}

在运行过程中，Q 表既参与调度决策，又在每次调度后进行更新，因此存在频繁的读写操作。为避免并发访问带来的数据竞争问题，本文在内存层面引入自旋锁对 Q 表进行保护，其主要策略包括：

\begin{itemize}
\item 在 Q 表打包（pack）与解包（unpack）过程中加锁，确保数据复制的原子性；
\item 在强化学习更新（Q 值更新）阶段加锁，避免并发修改同一状态-动作条目；
\item 在持久化读写过程中加锁，防止磁盘与内存数据不一致。
\end{itemize}

该设计基于 xv6 内核的轻量级并发模型，自旋锁适用于短时间临界区，能够在保证一致性的同时减少上下文切换开销。

在磁盘层面，文件系统通过 inode 锁与日志锁协同工作以保证数据一致性：inode 锁用于保护文件内容及元数据的并发访问，而日志机制（log）通过事务方式确保写入操作的原子性，即使在系统崩溃的情况下，也不会出现部分写入导致的数据不一致问题。

因此，Q 表在“内存—磁盘”两个层面均具备一致性保障，实现了从并发访问到持久化存储的完整可靠性设计。

\subsubsection{系统调用接口}

为提升系统的可操作性与调试能力，本文在内核中新增系统调用接口：

\begin{center}
\texttt{sys\_saveqtable()}
\end{center}

该接口用于触发 Q 表的持久化写入操作，其实现路径为：用户态程序通过系统调用进入内核态，调用 Q 表保存函数，完成从内存到磁盘的写入流程。

引入该接口具有以下几方面意义：

\begin{itemize}
\item 支持在实验过程中手动控制 Q 表保存时机，便于观察学习过程；
\item 避免仅依赖系统关闭或异常时写入带来的不确定性；
\item 为后续扩展（如周期性保存、策略导出等）提供接口基础。
\end{itemize}

通过该系统调用，强化学习调度模块从“内核内部机制”扩展为“可交互系统组件”，增强了整体系统的可调试性与可扩展性。

\subsection{状态空间与动作空间编码}

\subsubsection{全局状态离散化}

为了避免状态空间过大，本文并未直接使用单个进程的大量细粒度特征，而是从全局调度视角选择两个能够反映系统负载和等待压力的聚合指标：

\begin{itemize}
\item 当前处于 \texttt{RUNNABLE} 状态的进程数量；
\item 所有 \texttt{RUNNABLE} 进程中的最大等待时间。
\end{itemize}

其中，第一个指标反映当前可运行队列的拥塞程度，第二个指标反映系统中是否出现等待时间过长的进程。二者组合后，可以较好地描述调度器面临的整体局面，同时又保持状态维度较小。

在实现中，\texttt{state\_runnable\_bucket()} 将可运行进程数量划分为三个桶：

\begin{itemize}
\item 0 类：可运行进程数不大于 1；
\item 1 类：可运行进程数在 2 到 4 之间；
\item 2 类：可运行进程数大于 4。
\end{itemize}

\texttt{state\_maxwait\_bucket()} 则将最大等待时间划分为三个桶：

\begin{itemize}
\item 0 类：最大等待时间小于 5 个 tick；
\item 1 类：最大等待时间在 5 到 19 个 tick 之间；
\item 2 类：最大等待时间不小于 20 个 tick。
\end{itemize}

由于 \texttt{param.h} 中定义了 \texttt{NWAIT\_BUCKET = 3}，而 \texttt{rl.h} 中定义了 \texttt{NRUN\_BUCKET = 3}，因此总状态数为：

\begin{equation}
NSTATE = NRUN\_BUCKET \times NWAIT\_BUCKET = 3 \times 3 = 9
\end{equation}

状态编码公式为：

\begin{equation}
s = rb \times NWAIT\_BUCKET + wb
\end{equation}

其中，$rb$ 表示可运行进程数所在区间，$wb$ 表示最大等待时间所在区间。函数 \texttt{build\_global\_state()} 即通过遍历进程表统计上述信息，并返回最终状态编号。

\subsubsection{动作空间设计}

本实现中，动作并不表示某一个具体进程编号，而表示一种“选择进程的规则”。这种抽象方式能够让动作空间与系统当前进程数解耦，从而保证动作空间规模稳定。

结合 \texttt{proc.h} 中的定义，当前共有四个动作：

\begin{itemize}
\item \texttt{ACT\_PICK\_MAX\_WAIT}：优先选择等待时间最长的 \texttt{RUNNABLE} 进程；
\item \texttt{ACT\_PICK\_MIN\_SCHED}：优先选择被调度次数最少的进程；
\item \texttt{ACT\_PICK\_MIN\_RTIME}：优先选择累计运行时间最短的进程；
\item \texttt{ACT\_PICK\_RR}：按轮转调度规则选择下一个进程。
\end{itemize}

前三类动作属于带有明确优化倾向的启发式策略：最长等待优先更强调缓解饥饿，最少调度次数优先更强调调度公平性，最短累计运行时间优先则更偏向于补偿短作业。第四类动作为保留的传统 RR 规则，它既可作为基准行为，也为强化学习在早期探索阶段提供一个稳定且低风险的选择。

在 \texttt{pick\_proc\_by\_action()} 中，前三种策略通过遍历进程表并比较相应字段来选出最优候选者；如主比较条件相同，则再使用等待时间或调度次数作为次级比较依据。对于 RR 动作，则单独调用 \texttt{pick\_proc\_rr()}，依赖 \texttt{rr\_cursor} 在进程表中循环扫描，以保持传统轮转调度的行为特征。

\subsection{决策机制与学习更新}

\subsubsection{$\varepsilon$-greedy 动作选择}

在给定状态后，调度器通过 \texttt{choose\_action()} 完成动作选择。该函数采用 $\varepsilon$-greedy 策略平衡“利用当前最优经验”和“继续探索新动作”之间的关系。

代码中定义 \texttt{EPOSILON = 10}，并通过 \texttt{rand\_range(100)} 产生区间 \texttt{[0,100)} 内的随机数。因此：

\begin{equation}
\varepsilon = 0.1
\end{equation}

即系统以 10\% 的概率随机选择动作，以 90\% 的概率选择当前状态下 Q 值最大的动作。若状态编号越界，则函数默认返回动作 0，以保证实现上的安全性。

在贪心选择分支中，函数会遍历对应状态下的所有动作，选出 Q 值最大的动作编号；在探索分支中，则直接随机返回一个动作。这种实现方式简单直接，适合 xv6 这种资源有限、强调可读性的教学内核环境。

\subsubsection{随机数模块实现}

由于 xv6 原生内核并未提供可直接用于调度探索策略的随机数接口，本文增加了 \texttt{rand.c} 模块，为 $\varepsilon$-greedy 策略提供支持。该模块维护一个全局随机状态 \texttt{rand\_state}，并采用 xorshift 方式生成伪随机数。

该模块主要包含以下接口：

\begin{itemize}
\item \texttt{randinit(seed)}：初始化随机模块；
\item \texttt{srand(seed)}：重新设置种子；
\item \texttt{rand32()}：返回 32 位随机数；
\item \texttt{rand\_range(n)}：返回 \texttt{[0,n)} 范围内的随机整数。
\end{itemize}

为了避免多核并发访问导致随机状态损坏，\texttt{rand.c} 使用自旋锁 \texttt{rand\_lock} 保护 \texttt{rand\_state}。系统启动时，\texttt{main()} 在完成 \texttt{procinit()} 后调用 \texttt{qtableinit()} 和 \texttt{randinit(0)}，从而完成学习模块与随机模块的初始化。

\subsubsection{奖励函数设计}

本实现中的奖励函数采用相对简洁的设计，核心目标是鼓励调度器优先缓解等待压力，同时对“成功推动进程终止”给予额外正反馈。函数 \texttt{compute\_reward(old\_state, next\_state, terminal)} 的逻辑可以概括为：

\begin{itemize}
\item 若本轮调度结束后目标进程已成为 \texttt{ZOMBIE}，则奖励加 5；
\item 若新状态的最大等待时间桶编号低于旧状态，则奖励加 2；
\item 若新状态的最大等待时间桶编号高于旧状态，则惩罚减 2；
\item 若最大等待时间桶不变，则该部分奖励为 0。
\end{itemize}

这里使用的是“等待时间桶的变化”，而不是原始等待时间的精确差值。这样做的好处是与状态空间编码方式保持一致，能减少奖励信号的噪声，并降低实现复杂度。

从实现上看，\texttt{compute\_reward()} 并不直接使用旧状态中的可运行进程数量桶，而只提取状态编号中对应的等待桶编号进行比较。这意味着当前奖励设计主要优化的是“最大等待压力”，而不是系统吞吐、平均运行时间等其他指标。这样的设计取舍是有意为之，它与本文强调响应性和公平性的实验目标保持一致。

\subsubsection{Q 表更新}

当目标进程执行结束一个调度周期并返回调度器后，\texttt{scheduler()} 会重新构造后继状态 \texttt{next\_state}，然后调用 \texttt{rl\_update(state, action, next\_state, reward, terminal)} 更新 Q 表。

\texttt{rl\_update()} 的处理过程包括：

\begin{enumerate}
\item 检查状态和动作编号是否越界；
\item 读取旧值 \texttt{old\_q = qtable[s][a]}；
\item 在后继状态 \texttt{s'} 下扫描所有动作，求得 \texttt{max\_next}；
\item 若当前转移为终止情形，则目标值只取即时奖励；
\item 否则按 Bellman 目标值形式计算 \texttt{target}；
\item 按学习率更新 \texttt{qtable[s][a]}。
\end{enumerate}

其本质仍对应于标准 Q-learning 公式：

\begin{equation}
Q(s,a) \leftarrow Q(s,a) + \alpha \left( r + \gamma \max_{a'}Q(s',a') - Q(s,a) \right)
\end{equation}

只是在实际代码中用定点数和整数除法完成了等价近似。至此，调度器就形成了“观测状态、选择动作、执行调度、获取反馈、更新价值”的完整闭环。

\subsection{调度流程中的关键实现细节}

\subsubsection{调度主循环改造}

强化学习调度器的核心改动集中在 \texttt{scheduler()}。与 xv6 默认实现相比，新的调度主循环在选择进程前插入了状态构造和动作决策步骤，在进程返回后追加了奖励计算和价值更新步骤。

调度路径中的关键代码逻辑可概括为：

\begin{verbatim}
int state = build_global_state();
int action = choose_action(state);
best = pick_proc_by_action(action);
\end{verbatim}

若成功选中进程，则调度器在切换前还会维护若干统计字段：若 \texttt{first\_run\_time} 尚未记录，则以当前 \texttt{ticks} 作为首次运行时间；随后递增 \texttt{m\_sched\_count}，并记录 \texttt{m\_last\_scheduled\_tick}。之后将进程状态设置为 \texttt{RUNNING}，完成上下文切换。

进程返回后，调度器通过判断 \texttt{best->state == ZOMBIE} 来确定本次转移是否为终止情形，然后继续执行奖励计算与 Q 表更新。若当前没有可运行进程，则执行 \texttt{wfi} 指令等待中断到来。

\subsubsection{统计更新与时间基准}

强化学习模块能够工作，依赖于内核对时间和调度行为的持续记录。本实现选择将统计维护嵌入时钟中断路径。具体而言，在 \texttt{clockintr()} 中，在更新全局 \texttt{ticks} 后调用 \texttt{update\_sched\_stats()}。

该函数的更新规则如下：

\begin{itemize}
\item 当前 CPU 上若存在正在运行的进程且其状态为 \texttt{RUNNING}，则其 \texttt{m\_run\_ticks} 加一；
\item 对于其他所有 \texttt{RUNNABLE} 状态进程，其 \texttt{m\_wait\_ticks} 加一。
\end{itemize}

由于 xv6 的定时器中断是周期性发生的，因此这两类 tick 统计能够为后续的状态构造、奖励计算和实验评估提供统一时间基准。

\subsubsection{观测接口与调试输出}

为了支持实验阶段的数据采集，本文在原有 \texttt{wait()} 的基础上扩展实现了 \texttt{waitstat()} 系统调用，其内核实现为 \texttt{kwaitstat()}。当父进程等待子进程退出时，除原有退出状态外，还会把子进程的调度统计信息打包为 \texttt{struct pstat} 并复制到用户空间。

\texttt{pstat} 当前包含以下字段：

\begin{itemize}
\item \texttt{c\_time}；
\item \texttt{e\_time}；
\item \texttt{m\_run\_ticks}；
\item \texttt{m\_wait\_ticks}；
\item \texttt{m\_sleep\_ticks}；
\item \texttt{first\_run\_time}；
\item \texttt{m\_sched\_count}。
\end{itemize}

基于这些字段，用户态测试程序可以直接计算响应时间、周转时间、等待时间以及被调度次数等指标，为第五章的实验评估提供数据来源。

除了用户态接口外，本文还增强了 \texttt{procdump()} 的调试输出，使其能够打印每个活动进程的最近调度时刻、累计运行时间、累计等待时间、调度次数以及部分派生指标，并在输出末尾调用 \texttt{qtabledump()} 打印当前 Q 表内容。这一机制有助于在开发阶段观察学习过程是否按预期进行。

\subsection{本章小结}

本章围绕 xv6 内核中的 QL-Scheduler 落地实现，说明了强化学习调度器从算法设计到系统实现的具体映射关系。本文通过扩展 \texttt{struct proc} 维护运行统计信息，使用离散全局状态和有限动作集合构造 Q 表，在 \texttt{scheduler()} 中嵌入状态观测、动作选择、奖励反馈和 Q 值更新逻辑，并通过 \texttt{rand.c}、\texttt{waitstat()} 以及 \texttt{procdump()} 等辅助模块补齐探索与观测能力。

至此，第三章提出的状态空间、动作空间、奖励机制和学习策略已经在 xv6 中被具体实现为一个可在线运行的内核级调度原型。这也为下一章的实验设计与性能评估提供了完整的系统基础。
